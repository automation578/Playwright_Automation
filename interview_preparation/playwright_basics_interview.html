1. Playwright Introduction & Setup

Core concepts — the most common opener questions
1 What is Playwright? How is it different from Selenium?
Playwright is an open-source end-to-end testing framework by Microsoft that supports automation of Chromium, Firefox, and WebKit browsers. It was built from the ground up for modern web apps.

FeaturePlaywrightSelenium
CreatorMicrosoftOpen-source community
Auto-waitingBuilt-in — waits for element to be ready automaticallyManual — you write waits explicitly
Browser enginesChromium, Firefox, WebKit (Safari)Chrome, Firefox, Safari, Edge, IE
Language supportJS/TS, Python, Java, C#Java, Python, C#, JS, Ruby

#  - playwright.config.ts (configuration file)
#  - example tests in tests/ folder

# Run all tests
npx playwright test

# Run with UI (headed mode)
npx playwright test --headed

# Run specific file
npx playwright test tests/login.spec.ts

# Open Playwright's interactive UI mode
npx playwright test --ui
3 What browsers does Playwright support?
Playwright supports three browser engines — each covers multiple real browsers:
  • Chromium — covers Google Chrome and Microsoft Edge (both are Chromium-based)
  • Firefox — covers Mozilla Firefox
  • WebKit — covers Apple Safari (and iOS Safari behavior)
You can run the same test across all three browsers simultaneously using projects in playwright.config.ts.
4 What is the structure of a basic Playwright test?
// tests/login.spec.ts
import { test, expect } from '@playwright/test';

test('user can log in successfully', async ({ page }) => {
  // Navigate to the app
  await page.goto('https://example.com/login');

  // Fill in the form
  await page.fill('#username', 'admin');
  await page.fill('#password', 'Admin@123');

  // Click login button
  await page.click('button[type="submit"]');

  // Assert — verify landing page
  await expect(page).toHaveURL(/dashboard/);
  await expect(page.locator('h1')).toHaveText('Welcome, Admin');
});
Key concepts:
  • test() — declares a test case
  • { page } — fixture injected by Playwright (represents a browser tab)
  • async/await — all Playwright calls are asynchronous
  • expect() — used for assertions with built-in auto-waiting
5 What is a Playwright Fixture? What fixtures are available by default?
Fixtures are the objects injected into your test function via destructuring. They handle setup and teardown automatically.
FixtureTypeDescription
pagePageA single browser tab — the most used fixture
browserBrowserBrowser instance (Chrome, Firefox, WebKit)
contextBrowserContextIsolated session (cookies, localStorage) — like an incognito window
browserNamestringName of the current browser ('chromium', 'firefox', 'webkit')
requestAPIRequestContextFor making HTTP requests in API tests
You can also create custom fixtures using test.extend() to share setup logic across tests (e.g., a logged-in page fixture).
6 What is the difference between Page, BrowserContext, and Browser?
Think of it as a hierarchy:
  • Browser — The browser instance (e.g., Chrome). Created once, expensive to start.
  • BrowserContext — An isolated browsing session within the browser. Has its own cookies, localStorage, and session state. Like a fresh incognito window. Multiple contexts can run in parallel.
  • Page — A single browser tab within a context. One context can have multiple pages (tabs).
// Hierarchy example
const browser = await chromium.launch();
const context = await browser.newContext();   // isolated session
const page    = await context.newPage();      // a tab in that session
In Playwright Test runner: Each test gets its own fresh BrowserContext automatically — so tests are isolated by default. You don't need to manage this manually.

2. Locators — Finding Elements

The primary way Playwright finds elements on the page
7 What are Locators in Playwright? Why are they better than Selenium locators?
A Locator in Playwright is a lazy representation of an element (or set of elements). It doesn't look up the element immediately — it re-queries the DOM every time an action is performed on it.

Advantages over Selenium's WebElement:
  • Auto-waiting: Every action on a locator automatically waits for the element to be visible, enabled, and stable before acting
  • Auto-retrying: If an action fails (e.g., element not yet rendered), Playwright retries within the timeout
  • No StaleElementReferenceException: Because locators re-query the DOM on each use
  • Strict mode: If a locator matches multiple elements, Playwright throws an error, preventing silent bugs
const btn = page.locator('button#submit');
// 'btn' doesn't find the element yet

await btn.click();
// NOW it finds the element, waits for it to be ready, and clicks
8 What are the different ways to create locators in Playwright?
// 1. CSS Selector (most common)
page.locator('button.btn-primary')
page.locator('input[type="email"]')
page.locator('#username')

// 2. XPath
page.locator('xpath=//button[text()="Login"]')
page.locator('//input[@placeholder="Email"]')  // auto-detected as XPath

// 3. Role — preferred for accessibility and resilience
page.getByRole('button', { name: 'Submit' })
page.getByRole('textbox', { name: 'Email' })
page.getByRole('checkbox', { name: 'Remember me' })

// 4. Text content
page.getByText('Welcome back')
page.getByText('Submit', { exact: true })

// 5. Label — finds input associated with a label
page.getByLabel('Email address')
page.getByLabel('Password')

// 6. Placeholder
page.getByPlaceholder('Enter your email')

// 7. Alt text — for images
page.getByAltText('Company Logo')

// 8. Test ID — data-testid attribute (most stable)
page.getByTestId('submit-button')
// Matches: <button data-testid="submit-button">

// 9. Title attribute
page.getByTitle('Close dialog')
Playwright recommends this priority: getByRole → getByLabel → getByPlaceholder → getByText → getByTestId → CSS/XPath. Role-based locators are most resilient to UI changes.
9 How do you filter, chain, and narrow down locators?
// Scope locator within a parent element
const form = page.locator('form#login');
const emailField = form.locator('input[type="email"]'); // scoped to form

// Filter by text
page.locator('li').filter({ hasText: 'Playwright' })

// Filter by child element
page.locator('tr').filter({ has: page.locator('td:has-text("John")') })

// nth() — get specific element by index (0-based)
page.locator('ul li').nth(0)    // first item
page.locator('ul li').last()    // last item
page.locator('ul li').first()   // first item

// and() — combine locators
page.locator('button').and(page.getByText('Submit'))

// or() — element matching either condition
page.getByRole('button').or(page.getByRole('link'))
10 What is the difference between page.locator() and page.$() / page.$()?
  • page.locator() — Modern, recommended. Returns a Locator object with auto-waiting, auto-retrying, and strict mode. Re-queries DOM on each action.
  • page.$() — Legacy API. Returns an ElementHandle (like Selenium's WebElement) — a snapshot reference that can go stale. Does NOT auto-wait. Avoid in new code.
  • page.$$() — Legacy API. Returns ElementHandle[]. Same limitations as page.$().
Avoid page.$() and page.$$() in new Playwright code. Always prefer page.locator() and its family (getByRole, getByText, etc.).

3. Actions — Interacting with Elements

Click, fill, type, select, drag — every common interaction
11 What are the most common Playwright actions?
// NAVIGATION
await page.goto('https://example.com');
await page.goBack();
await page.goForward();
await page.reload();

// CLICK
await page.click('#submit');
await page.locator('button').click();
await page.locator('button').dblclick();          // double-click
await page.locator('div').click({ button: 'right' }); // right-click

// FILL / TYPE (for text inputs)
await page.fill('#email', 'test@example.com');    // clears and fills instantly
await page.type('#email', 'test@example.com');    // types character by character
await page.locator('#search').clear();             // clear input
await page.locator('#search').fill('Playwright');

// KEYBOARD
await page.press('#input', 'Enter');
await page.keyboard.press('Tab');
await page.keyboard.type('Hello World');
await page.keyboard.down('Shift');
await page.keyboard.up('Shift');

// SELECT DROPDOWN
await page.selectOption('#country', 'US');              // by value
await page.selectOption('#country', { label: 'United States' }); // by label
await page.selectOption('#country', { index: 2 });       // by index

// CHECKBOX / RADIO
await page.check('input[name="agree"]');
await page.uncheck('input[name="agree"]');
await page.locator('#terms').setChecked(true);   // explicit true/false

// HOVER
await page.hover('.menu-item');

// DRAG AND DROP
await page.dragAndDrop('#source', '#target');

// SCROLL
await page.locator('#element').scrollIntoViewIfNeeded();
await page.mouse.wheel(0, 300); // scroll down 300px

// READ DATA
const text  = await page.locator('h1').textContent();
const inner = await page.locator('div').innerText();
const value = await page.locator('input').inputValue();
const attr  = await page.locator('a').getAttribute('href');
12 What is the difference between fill() and type()?
  • fill(selector, value) — Clears the field and sets the value instantly (no simulated keystrokes). Faster and preferred for most tests.
  • type(selector, value) — Simulates pressing each key one by one, including triggering keydown, keypress, keyup, and input events per character. Use this when the app has autocomplete/typeahead or character-by-character event listeners.
// Use fill() — fast, standard
await page.fill('#search', 'Playwright testing');

// Use type() — when app needs real keystroke events
await page.locator('#autocomplete').type('New Y');
// wait for dropdown to appear, then click option
await page.getByText('New York').click();
13 How do you take a screenshot and record a video in Playwright?
// Full page screenshot
await page.screenshot({ path: 'screenshots/home.png', fullPage: true });

// Element screenshot
await page.locator('#chart').screenshot({ path: 'screenshots/chart.png' });

// Screenshot on failure — configure in playwright.config.ts
// playwright.config.ts — auto screenshot and video
export default defineConfig({
  use: {
    screenshot: 'only-on-failure',  // 'on', 'off', 'only-on-failure'
    video: 'retain-on-failure',      // 'on', 'off', 'retain-on-failure'
    trace: 'on-first-retry',          // captures trace for debugging
  }
});
Trace Viewer: Playwright's trace captures a full timeline of your test — screenshots at each step, network requests, console logs. View it with npx playwright show-trace trace.zip.

4. Assertions

expect() — Playwright's built-in assertion library with auto-waiting
14 What types of assertions does Playwright provide?
Playwright has two types of assertions — web-first (preferred) and generic. 
// ── WEB-FIRST ASSERTIONS (auto-wait until condition is true) ──

// Page-level
await expect(page).toHaveURL('https://example.com/dashboard');
await expect(page).toHaveURL(/dashboard/);       // regex match
await expect(page).toHaveTitle('Dashboard');
await expect(page).toHaveTitle(/Dashboard/);

// Locator-level
await expect(page.locator('h1')).toHaveText('Welcome');
await expect(page.locator('h1')).toContainText('Welc');
await expect(page.locator('#error')).toBeVisible();
await expect(page.locator('#spinner')).toBeHidden();
await expect(page.locator('button')).toBeEnabled();
await expect(page.locator('button')).toBeDisabled();
await expect(page.locator('#agree')).toBeChecked();
await expect(page.locator('input')).toHaveValue('admin@test.com');
await expect(page.locator('div')).toHaveAttribute('class', 'active');
await expect(page.locator('li')).toHaveCount(5);

// Negative assertions — prefix with .not
await expect(page.locator('#error')).not.toBeVisible();
await expect(page).not.toHaveURL(/login/);

// ── GENERIC (value / boolean assertions, no auto-wait) ──
expect(count).toBe(5);
expect(title).toContain('Dashboard');
expect(items).toHaveLength(3);
expect(obj).toBeTruthy();
expect(obj).toBeNull();
15 What is the difference between toHaveText() and toContainText()?
  • toHaveText('Welcome, Admin') — The element's text must exactly match the given string (after normalization of whitespace).
  • toContainText('Welcome') — The element's text must contain the given substring. More flexible.
// Element: <h1>Welcome, Admin</h1>
await expect(page.locator('h1')).toHaveText('Welcome, Admin');  // ✓ passes
await expect(page.locator('h1')).toHaveText('Welcome');          // ✗ fails
await expect(page.locator('h1')).toContainText('Welcome');       // ✓ passes
16 What is a soft assertion in Playwright? How do you use it?
By default, a failed assertion stops the test immediately. Soft assertions collect all failures and report them all at the end. 
import { test, expect } from '@playwright/test';

test('validate user profile page', async ({ page }) => {
  await page.goto('/profile');

  // Soft assertions — test continues even if one fails
  await expect.soft(page.locator('#name')).toHaveText('John Doe');
  await expect.soft(page.locator('#email')).toHaveText('john@test.com');
  await expect.soft(page.locator('#avatar')).toBeVisible();

  // Test is marked failed if any soft assertion above failed,
  // but all assertions were still evaluated
});
When to use: Validating multiple independent fields on a single page. When failure of one field shouldn't stop checking others (e.g., user profile validation, data table row verification).

5. Waits & Auto-Waiting

How Playwright eliminates the need for manual waits
17 What is Auto-waiting in Playwright?
Playwright's biggest advantage — every action on a locator automatically waits for the element to be in an actionable state before executing. No explicit waits needed for standard interactions.

Before performing an action, Playwright checks:
  • Visible — element is in the DOM and not hidden
  • Stable — element is not animating or moving
  • Enabled — element is not disabled
  • Editable — (for type/fill) element can receive input
  • Receives events — no other element is overlaying it
// Playwright automatically waits before each action below
await page.click('#submit');      // waits until visible, stable, enabled
await page.fill('#email', 'x');  // waits until editable

// You don't need this anymore (unlike Selenium):
// await page.waitForSelector('#submit'); // usually not needed
18 When do you still need explicit waits in Playwright?
Despite auto-waiting, some scenarios need explicit waits: 
// 1. Wait for URL change (after navigation)
await page.waitForURL('**/dashboard');
await page.waitForURL(/dashboard/);

// 2. Wait for an element to appear
await page.waitForSelector('#result-table');

// 3. Wait for an element to disappear (e.g., loading spinner)
await page.locator('#spinner').waitFor({ state: 'hidden' });

// 4. Wait for network request to complete
await page.waitForResponse('**/api/users');

// 5. Wait for page load state
await page.waitForLoadState('networkidle'); // no network for 500ms
await page.waitForLoadState('domcontentloaded');

// 6. Custom wait — wait for function to return true
await page.waitForFunction(() => document.querySelectorAll('tr').length > 5);

// 7. Specific element state
await page.locator('#btn').waitFor({ state: 'visible' });
// States: 'attached' | 'detached' | 'visible' | 'hidden'
Avoid page.waitForTimeout(2000) — this is a hardcoded sleep. Only use for debugging, never in production tests.
19 What is the default timeout in Playwright and how do you change it?
Timeout TypeDefaultApplies to
Action timeout0 (uses test timeout)Individual click/fill/etc.
Test timeout30,000ms (30s)Each test function
Assertion timeout5,000ms (5s)expect() calls
Navigation timeout30,000ms (30s)goto(), waitForURL()
// playwright.config.ts — global timeouts
export default defineConfig({
  timeout: 60_000,          // test timeout: 60 seconds
  expect: {
    timeout: 10_000         // assertion timeout: 10 seconds
  },
  use: {
    actionTimeout: 15_000,  // per-action timeout
    navigationTimeout: 30_000
  }
});

// Per-test override
test('slow test', async ({ page }) => {
  test.setTimeout(120_000);  // override for this test only
  ...
});

// Per-action timeout
await page.click('#btn', { timeout: 5000 });
await expect(page.locator('#msg'), { timeout: 10000 }).toBeVisible();

6. Handling Web Scenarios

Dialogs, iFrames, multiple tabs, file upload/download
20 How do you handle JavaScript dialogs (alert, confirm, prompt)?
// IMPORTANT: Register dialog handler BEFORE the action that triggers it

// Accept an alert
page.on('dialog', async dialog => {
  console.log(dialog.message());  // get alert text
  await dialog.accept();            // click OK
});
await page.click('#delete-btn');   // triggers alert

// Dismiss a confirm dialog (click Cancel)
page.on('dialog', dialog => dialog.dismiss());

// Handle a prompt (enter text then accept)
page.on('dialog', dialog => dialog.accept('My answer'));

// One-time handler using page.once()
page.once('dialog', dialog => dialog.accept());
Key difference from Selenium: Playwright uses an event-driven approach. You register a listener that fires when a dialog appears — no need to switch context.
21 How do you handle iFrames in Playwright?
// Use frameLocator() — no context switching needed!
const frame = page.frameLocator('iframe#payment');

// Interact with elements inside the iframe
await frame.locator('input[name="cardNumber"]').fill('4111111111111111');
await frame.getByLabel('Expiry').fill('12/28');
await frame.getByRole('button', { name: 'Pay' }).click();

// Nested iframes
const innerFrame = page
  .frameLocator('iframe.outer')
  .frameLocator('iframe.inner');
await innerFrame.locator('button').click();
Much simpler than Selenium: Playwright's frameLocator() creates a scoped locator — no need to switchTo().frame() and switchTo().defaultContent() back and forth.
22 How do you handle multiple browser tabs or windows?
// Listen for new page (tab/window) event
const [newPage] = await Promise.all([
  context.waitForEvent('page'),           // wait for new tab to open
  page.click('a[target="_blank"]')         // click that opens new tab
]);

// Wait for new page to load
await newPage.waitForLoadState();
console.log(await newPage.title());

// Interact with the new tab
await newPage.locator('#proceed').click();

// Close new tab and continue on original page
await newPage.close();
// 'page' still refers to the original tab

// Open a new tab programmatically
const tab2 = await context.newPage();
await tab2.goto('https://example.com');
23 How do you handle file upload and download?
// ── FILE UPLOAD ──
// Method 1: setInputFiles — for <input type="file">
await page.setInputFiles('input[type="file"]', './files/document.pdf');

// Multiple files
await page.setInputFiles('input[type="file"]', ['file1.pdf', 'file2.png']);

// Clear file input
await page.setInputFiles('input[type="file"]', []);

// Method 2: filechooser event — for custom upload buttons

await fileChooser.setFiles('./files/document.pdf');

// ── FILE DOWNLOAD ──

// Save to disk

24 How do you intercept and mock network requests?
// Block all image requests (speed up tests)
await page.route('**/*.{png,jpg,jpeg,gif,svg}', route => route.abort());

// Mock an API response
await page.route('**/api/users', async route => {
  await route.fulfill({
    status: 200,
    contentType: 'application/json',
    body: JSON.stringify([{ id: 1, name: 'Mock User' }])
  });
});

// Modify a real request (add header)
await page.route('**/api/**', async route => {
  const headers = { ...route.request().headers(), 'x-custom': 'test' };
  await route.continue({ headers });
});

// Wait for a specific request to be made
const [response] = await Promise.all([
  page.waitForResponse('**/api/login'),
  page.click('#login-btn')
]);
console.log(await response.json());

7. Page Object Model (POM) in Playwright

Organizing tests with reusable page classes
25 How do you implement POM in Playwright?
// pages/LoginPage.ts
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  readonly page: Page;
  readonly usernameInput: Locator;
  readonly passwordInput: Locator;
  readonly loginButton:   Locator;
  readonly errorMessage:  Locator;

  constructor(page: Page) {
    this.page          = page;
    this.usernameInput = page.getByLabel('Username');
    this.passwordInput = page.getByLabel('Password');
    this.loginButton   = page.getByRole('button', { name: 'Login' });
    this.errorMessage  = page.locator('.error-message');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(username: string, password: string) {
    await this.usernameInput.fill(username);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }

  async getErrorMessage(): Promise<string> {
    return await this.errorMessage.innerText();
  }
}
// tests/login.spec.ts — test using POM
import { test, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

test('valid login redirects to dashboard', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('admin', 'Admin@123');
  await expect(page).toHaveURL(/dashboard/);
});

test('invalid login shows error', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('wrong@test.com', 'badpass');
  await expect(loginPage.errorMessage).toBeVisible();
  await expect(loginPage.errorMessage).toContainText('Invalid credentials');
});
26 How do you share a logged-in state across tests using fixtures?
// fixtures/auth.ts — save auth state once, reuse everywhere
import { test as base, Page } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

type Fixtures = { authenticatedPage: Page };

export const test = base.extend<Fixtures>({
  authenticatedPage: async ({ page }, use) => {
    // Setup: login once
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await loginPage.login('admin', 'Admin@123');

    // Provide to test
    await use(page);

    // Teardown: runs after test (optional)
  }
});

// Use in test:
test('can see dashboard', async ({ authenticatedPage }) => {
  await expect(authenticatedPage.locator('h1')).toContainText('Dashboard');
});
Even better performance: Use storageState in playwright.config.ts to save auth cookies/localStorage to a file and reuse across all tests without re-logging in.

8. Playwright Configuration & Test Runner

playwright.config.ts — hooks, parallelism, projects, reporters
27 Explain the key sections of playwright.config.ts.
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',              // where to look for test files
  testMatch: '**/*.spec.ts',       // file pattern
  timeout: 30_000,                  // test timeout (30s)
  retries: 2,                       // retry failed tests 2 times
  workers: 4,                       // parallel workers (4 tests at a time)
  fullyParallel: true,              // run tests in each file in parallel
  reporter: [
    ['html'],                        // built-in HTML report
    ['list'],                        // console output
  ],

  use: {
    baseURL: 'https://staging.example.com',  // page.goto('/login') = baseURL+'/login'
    headless: true,
    screenshot: 'only-on-failure',
    video:      'retain-on-failure',
    trace:      'on-first-retry',
    actionTimeout: 10_000,
  },

  // Run same tests on multiple browsers
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome']  } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit',   use: { ...devices['Desktop Safari']  } },
    // Mobile testing
    { name: 'mobile-chrome', use: { ...devices['Pixel 5'] } },
    { name: 'mobile-safari', use: { ...devices['iPhone 13'] } },
  ],
});
28 What are test hooks in Playwright?
import { test, expect } from '@playwright/test';

// Runs once before ALL tests in this file
test.beforeAll(async ({ browser }) => {
  console.log('Suite starting');
});

// Runs before EACH test
test.beforeEach(async ({ page }) => {
  await page.goto('/home');
});

// Runs after EACH test
test.afterEach(async ({ page }) => {
  // page is automatically closed after each test anyway
});

// Runs once after ALL tests
test.afterAll(async () => {
  console.log('Suite done');
});

// Group tests with describe
test.describe('Login feature', () => {
  test.beforeEach(async ({ page }) => {
    await page.goto('/login');
  });

  test('valid login', async ({ page }) => { ... });
  test('invalid login', async ({ page }) => { ... });
});
29 How do you skip, focus, and tag tests?
// Skip a test
test.skip('not implemented yet', async ({ page }) => { ... });

// Skip conditionally
test('safari only test', async ({ page, browserName }) => {
  test.skip(browserName !== 'webkit', 'Only for Safari');
  ...
});

// Run ONLY this test (focus mode)
test.only('debug this test', async ({ page }) => { ... });

// Mark as flaky — retries but doesn't fail suite if it fails
test.fixme('known issue', async ({ page }) => { ... });

31 How do you make API requests in Playwright?
import { test, expect } from '@playwright/test';

// GET request
test('GET users returns 200', async ({ request }) => {
  const response = await request.get('https://reqres.in/api/users?page=1');
  expect(response.status()).toBe(200);
  const body = await response.json();
  expect(body.data).toHaveLength(6);
});

// POST request
test('POST creates a user', async ({ request }) => {
  const response = await request.post('https://reqres.in/api/users', {
    data: { name: 'John QA', job: 'Tester' }
  });
  expect(response.status()).toBe(201);
  const body = await response.json();
  expect(body.name).toBe('John QA');
  expect(body.id).toBeTruthy();
});

// PUT / PATCH / DELETE
await request.put(`/api/users/${id}`, { data: { job: 'Lead' } });
await request.patch(`/api/users/${id}`, { data: { name: 'Updated' } });
await request.delete(`/api/users/${id}`);

34 How do you handle a test that needs to be logged in before running?
// Step 1: Create a setup script to save auth state
// tests/auth.setup.ts
import { test as setup } from '@playwright/test';

setup.describe('authenticate', () => {
  setup('save user session', async ({ page }) => {
    await page.goto('/login');
    await page.fill('#username', 'admin');
    await page.fill('#password', 'Admin@123');
    await page.click('#login-btn');
    // Save cookies + localStorage to file
    await page.context().storageState({ path: 'auth/admin.json' });
  });
});
// Step 2: playwright.config.ts — run auth before tests, reuse state
projects: [
  { name: 'setup', testMatch: '**/auth.setup.ts' },
  {
    name: 'logged-in tests',
    dependencies: ['setup'],
    use: { storageState: 'auth/admin.json' }  // auto-injects saved auth
  }
]

// Result: login happens ONCE; all tests in 'logged-in tests' start pre-authenticated
35 How do you read data from a table in Playwright?
// Count rows
const rowCount = await page.locator('table tbody tr').count();
console.log(`Total rows: ${rowCount}`);

// Get all cell texts in a column
const names = await page.locator('table tbody tr td:first-child').allTextContents();
// Returns: ['Alice', 'Bob', 'Charlie']

// Find a row containing "John" and click Edit in that row
const row = page.locator('tr').filter({ hasText: 'John' });
await row.getByRole('button', { name: 'Edit' }).click();

// Get specific cell value
const price = await page.locator('table tbody tr:nth-child(2) td:nth-child(3)').innerText();

// Verify all items in a list
await expect(page.locator('table tbody tr')).toHaveCount(5);
36 How do you use Playwright's codegen to generate tests?
Playwright Codegen records your browser actions and generates test code automatically. 
# Open browser with code recorder
npx playwright codegen https://example.com

# Record with specific browser
npx playwright codegen --browser=firefox https://example.com

# Record and save to file
npx playwright codegen --output=tests/generated.spec.ts https://example.com

# Record with specific viewport
npx playwright codegen --viewport-size=1280,720 https://example.com
Usage: Codegen is great for quickly generating locators and understanding what actions to use. Always review and refactor the generated code — it may not use the best locators (e.g., it may generate XPath when a role locator would be better).
37 Quick-fire: 10 short Playwright questions
  1. What language does this project use? — JavaScript and TypeScript (officially also Python, Java, C#)
  2. What is the default test file pattern? — *.spec.ts or *.test.ts
  3. How do you check if an element exists without failing? — await page.locator('#el').count() > 0 or isVisible()
  4. How do you get the text of all matching elements? — await page.locator('li').allTextContents()
  5. What is the Playwright Trace Viewer? — A GUI tool that shows a step-by-step timeline of a test — screenshots, DOM snapshots, network requests, console logs.
  6. What does fullyParallel: true do? — Runs tests within each file in parallel (not just across files).
  7. How do you emulate a mobile device? — Use devices['iPhone 13'] in projects config: use: { ...devices['iPhone 13'] }
  8. What is the difference between page.click() and locator.click()? — page.click(selector) is a shorthand that internally creates a locator. page.locator(selector).click() is equivalent and preferred for reusability.
  9. How do you check if a checkbox is checked? — await expect(page.locator('#agree')).toBeChecked()
  10. What command opens the Playwright HTML report? — npx playwright show-report
